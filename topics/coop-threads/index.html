1. Introduction

In this short note, we look at how continuations, in particular, the let/cc construct in Racket/Scheme may be used to implement a simple scenario of cooperating threads. The example is adapted from the Scheme example in the Wikipedia article on Continuations. Note: Racket has its own native implementation of threads, which are much more robust. We will not rely on that implementation. Instead we will build our own, but rely on Racket's built-in let/cc.

2. Problem specification: Ping Pong

The scenario consists of two threads, ping, and pong, each of which are infinite loops that print their identity and the value of a counter. The control alternates between ping and pong. ping prints a value, then prompts at for a command to continue or quit. A quit returns to the top level. On anything else, ping yields control to a waiting thread, most likely pong which then proceeds to execute from where it left off, and the cycle continues forever.

A thread is modeled as a thunk that, when invoked, potentially never terminates. 

;;; ping-pong.rkt
;;; =============
(provide
go
)
(require "scheduler.rkt")

(define done (lambda (arg) "unitialized"))

;;; done now holds the continuation that escapes to top level.
(let/cc k
   (set! done k))

(define (make-thread who)
  (lambda ()
    (letrec ([loop (lambda (n)
                     (printf "~a ~a\n" who n)
                     (if (eq? (read) 'quit)
                       (begin
                         (printf "quitting!~n")
                         (done)
                       )
                       (begin 
                         (yield)
                         (loop (add1 n)))))])
      (loop 0))))

(define ping (make-thread "ping"))
(define pong (make-thread "pong"))

3. Getting it going

Getting the threads going involves three steps: initializing the scheduler, registering the threads with a scheduler, and starting the scheduler. 

(define (go)
     (init)
     (register ping)
     (register pong)
     (start))

The ping-pong scenario plays out as in the following example of a run: 

> (go)
ping 0
OK
pong 0
OK
ping 1
FINE
pong 1
ok
ping 2
quit
quitting!
>

4. Implementation

The implementation is in three parts. The definition of the threads themselves and a protocol to get everything started, which we have already seen. A scheduler library for thread life-cycle management and scheduling provided by three functions: register, start and yield, and a simple queue data structure used by the scheduler.

5. Queue

The queue data structure is inspected and manipulated using three functions: init-queue, empty-queue?, enqueue and dequeue: 

;;; queue.rkt
;;; =========
(provide
  init-queue
  empty-queue?
  enqueue
  dequeue
)

(define *queue* '())

(define (init-queue)
  (set! *queue* '()))

(define (empty-queue?)
  (null? *queue*))

(define (enqueue x)
  (set! *queue* (append *queue* (list x))))

(define (dequeue)
  (let ((x (car *queue*)))
    (set! *queue* (cdr *queue*))
    x))

6. Scheduler

The scheduler exposes four functions:

  • init initializes the scheduler.
  • register takes a thread and simply enqueues it for further execution.
  • start runs the next thread in the queue
  • yield captures the current control state, enqueues it, and proceeds to execute the next thread on the queue.
;;; scheduler.rkt
;;; =============
(provide
  init
  register
  start
  yield)

(require "queue.rkt")

;;; initialize the scheduler
(define (init)
  (init-queue))

;;; registers thread for execution
(define (register thread)
  (enqueue thread))

;; picks the next thread in the queue and runs it
(define (start)
  (if (empty-queue?)
    (printf "no threads in queue: done!~n")
    ((dequeue))))

(define (yield)
  (let/cc k
    (enqueue k)
    ((dequeue))))

7. Conclusion

This simple scenario illustrates the point that let/cc is a powerful construct that can be used to implement a non-trivial control scenario involving multiple threads. There are several design and performance issues around let/cc which make it less attractive to implement threads in terms of it, but we shall defer their discussion. Interested readers may refer to Oleg Kisleyov's article: An argument against call/cc.
