1. Introduction

In this interpreter, we shall define a monad data type, along with the unit and composition operators to model exceptions. Again, the term structure of the language stays the same: 

#lang racket
(provide (all-defined-out))

(require eopl/eopl)
(define-datatype term term?
  [con (int integer?)]
  [div (num term?) (den term?)])

2. Exceptionful computation

The value? and exception? type remain the same: 

;;; Semantic Domains
(define value? integer?)
(define exception? string?)

The exceptionful computation is represented by a monad (synonymous with computation). 

;;; computation? = (raise exception?) | (return value?)
(define-datatype computation computation?
  [raise (ex exception?)]
  [return (ans value?)])

3. Evaluator

We now derive the monadic version of the evaluator. We start with the evaluator in v1-exceptions: 

 1: ;;; eval-term : term? -> computation?
 2: (define eval-term
 3:   (lambda (t)
 4:     (cases term t
 5:       [con (int) (return int)]
 6:       [div (num den)
 7:         (let 
 8:           ([cn (eval-term num)])    ; (eval-term-num)
 9:           (cases computation cn
10:             [raise (ex) (raise ex)]
11:             [return (n)  
12:               (let ; (receiver-n)
13:                 ([cd (eval-term den)])   ; (eval-term-den)
14:                 (cases computation cd
15:                   [raise (ex) (raise ex)]
16:                   [return (d)
17:                     (if (= d 0)   ; (receiver-d)
18:                       (raise "divide by zero")
19:                       (return (quotient n d)))]))]))])))

To convert the above interpreter to monadic form, we need to do three things: identify unit, the composition operator o and the receiver.

4. Identifying the unit operator

The unit operator maps a value to a computation. This is already being done by the return constructor. 

1: ;;; unit : value? -> computation?
2: (define unit return)

5. Identifying the composition operator o

Note the sub-computations on lines eval-term-num and eval-term-den. The corresponding receivers need to be constructed as functions from the expressions on lines receiver-n and receiver-d. To define the composition operator, we try to abstract everything that is between line eval-term-num and receiver-n. This gives us a plausible definition of compostion: 

;;; o : computation? -> [(value? -> computation?) -> compuatation?]
(define o
  (lambda (m)
    (lambda (k)
      (cases computation m
        [raise (ex) (raise ex)]
        [return (n)
          (k n)]))))

6. Rewriting the evaluator with unit and composition.

Using the definitions of unit and compostion o, and curried application, we rewrite the evaluator: 

 1: ;;; : : [['a -> ['b -> 'c]] 'a 'b] -> 'c
 2: (define :
 3:   (lambda (f v1 v2)
 4:     ((f v1) v2)))
 5: 
 6: ;;; eval-term : term? -> computation?
 7: (define eval-term
 8:   (lambda (t)
 9:     (cases term t
10:       [con (int) (unit int)]
11:       [div (num den)
12:         (: o (eval-term num)
13:           (lambda (n)
14:             (: o (eval-term den)
15:               (lambda (d)
16:                 (if (= d 0)
17:                   (raise "divide by zero")
18:                   (unit (quotient n d)))))))])))

7. Unit Tests

;;; Testing

;;; Racket's unit testing framework
(require rackunit)

(check-equal? (eval-term (con 5)) (return 5) "eval-term: 5 test")
(check-equal? (eval-term (div (con 5) (con 3))) (return 1) "eval-term: 5 test")
(check-equal? (eval-term (div (con 5) (con 0))) (raise "divide by zero") "eval-term: divide by zero test")

Notice that the last test case, the result of the computation is a 'raised' exception, while in the first two test cases, the results are 'return' values.

8. Summary

Exceptions and values are two variants of a computation. The interpreter maps terms to computations.
