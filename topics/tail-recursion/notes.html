1. Recursion - revisited

(letrec ([f (lambda (n)
            (if (zero? n) 1 (f (- n 1))))]
          [y 3])
     (f 1))

G = global environment

ε = {f|->C, y|->3}

C = <(n), (if (zero? n) 1 (f ( - n 1))), ε.G}

σ = { n|-> 1}

  • f is bound to a closure c in the environment ε
  • f gets evaluated in the environment where the arguments are bound to values (σ) and the environment in which it was declared (G)
  • difference between let and letrec
    • environment under which c is evaluated is G in let while it is ε.G for letrec
    • letrec is characterised by the circularity in the structures closure and environment
  • THUNK - (λ() e)
    • parameterless closure
    • to defer the evaluation of any expression
  • how values get transported
    • passed to the parent for further evaluation
    • environment: top-down & values: bottom-up
    • exception: recursion
      • value returned to the place where it is called from
      • hence the child tree needs to remember where the answer has to be fed
      • needs stack structure

2. Tail Recursion

  • It is a special type of recursion.
  • It has a concept of tail call.
  • A tail call is when the last call to a function returns the answer to the entire recursive block
  • You can identify the tail call just by seeing the program.
  • Consider the below code and its AST evaluation.
> (letrec ([f (lambda (n) 
             (if (zero? n) 1 (f (- n 1))))] 
           [y 3])
    (let ([n 5])
      (f 1)))
  • Corresponding AST evaluation figure
  • Observe how the calulated values move up the tree.
  • The end value that was calculated propagates up the tree without any modification i.e no other calculations need performed on the result.
  • This happens because f is the tail call in the body.
  • The result of letrec is the result of (f 1) the result of f is the result of the if statement and the result of the if statement is either 1 or result of the function f
  • Consider the below code which is not a tail call
> (let ([f (lambda (x)
       		(+ 1 (g (sub1 x))))])
       (f 3))
  • Here g is another function available in the environment and (f 3) is equivalent to (+ 1 (g 2)) , thus calculating (g 2) is not sufficient the result must be incremented by 1 before propagating the result up the tree.
  • And thus is not the tail call
  • If the program was as below then it would have been a tail call
> (let ([f (lambda (x)
       (g (sub1 x)))]) (f 3))
  • Here (f 3) is equivalent to (g 2) and no other changes to the result of (g 2) need to be done to get the final answer and hence is a tail call.
  • The factorial code below is also not tail recursive
> (letrec ([f (lambda (n)
                (if (= n 0) 
                    1 
                    (* n (f (- n 1)))))])
    (f 1))
  • Here the result of f is the result of if and the result of if is either 1 or n multiplied with result of f.
  • This program is traced as shown below for the (f 3)

(f 3)	= (* 3 (f 2))
	= (* 3 (* 2 (f 1)))
	= (* 3 (* 2 (* 1 (f 1))))
	= (* 3 (* 2 (* 1 1)))
	= (* 3 (* 2 1))
	= (* 3 2)
	= 6
  • notice how the result is modified at each stage.
  • The next section explains how to write tail recursive functions using accumulator.

3. Using Accumulator

  • The below code shows how the factorial program can be made tail recursive using an accumulator.
(define !
	(lambda (n) (g n 1)))

(define g
	(lambda ( i a )
		(if (= i 0) a
			(g (sub1 i) (* i a)))))
  • The below shows how the above program is traced for (f 3)

(f 3)   = (g 3 1)
        = ((lambda (i a) (..)) 3 1) = (if (= 3 0) 1 (g 2 3))
        = (g 2 3)
        = (g 1 6)
        = (g 0 6)
        = 6
  • notice how the end answer turn out to be the final answer.
  • any program can be converted into a tail recursion by using accumulator and splitting the function into two sub-functions
  • tail recursion is equivalent to iteration
