1. Introduction

These notes present an interpreter for a language with first class functions

The purpose of this note is to examine transformations between different representations of continuations as they occur in the interpreter for a small language consisting of constants, (global) identifiers and `if' expressions. The goal is to realise an interpreter that is a transition system.

We start with a standard `direct style' interpreter. We then convert the interpreter into continuation passing style. Next, we convert the interpreter to imperative form with registers and a stepper. All continuations are now thunks (functions of zero arguments) and function calls in the interepreter are replaced by `goto's. The program counter points to the current (tail recursive) function to be executed. Each function call updates a set of global variables. What is interesting about this is that the program counter assumes values that are dynamically generated in the heap (due to the creation of continuation closures at rutime).

The second set of transformations start by converting the CPS interpreter to a (continuation) representation independent form. From representation independent form, continuations are represented as variant records. With this transformation, the interpreter consists of three main functions: eval-ast, lookup-env and apply-k and the program counter is limited to them and no longer points to the heap. The final transformation of turning the interpreter into imperative form with continuations as record data is left as an exercise.

2. A small language

2.1. Abstract Syntax

To keep the example tractable, we work with a small language of numbers, booleans, identifiers and if expressions. 

;;; Syntax
;;; <exp> ::= <num> | <bool> | <id-ref>
;;;           (ifte <exp> <exp> <exp>)

The racket implementation is given below: 

(require eopl)

(define-datatype ast ast?
  [num (n number?)]
  [bool (b boolean?)]
  [id-ref (x symbol?)]
  [ifte (test ast?) (then ast?)
        (else-ast ast?)])

along with test cases: 

(require rackunit)
(check-true (ast? (num 6)))
(check-true (ast? (bool #t)))
(check-true (ast? (id-ref 'x)))
(check-true (ast? (ifte (bool #t) (num 6) (id-ref 'x))))

2.2. Parser

The parser take 

;;; exp? are Racket s-expressions. 
(define exp? any/c)

;;; parse : datum? -> exp? + void?
;;; Here void? is what is `returned' where there is an error
(define (parse exp)
  (match exp
    [(? number? n) (num n)]
    [(? boolean? b) (bool b)]
    [(? symbol? x) (id-ref x)]
    [(list 'ifte a b c) (ifte (parse a) (parse b) (parse c))]
    [else (error 'parse "invalid syntax: ~a" exp)]))

(check-equal? (parse '5) (num 5))
(check-equal? (parse #t) (bool #t))
(check-equal? (parse 'x) (id-ref 'x))
(check-equal? (parse '(ifte #t 3 x)) (ifte (bool #t) (num 3) (id-ref 'x)))

3. Semantic Domains and Environments

3.1. Values

A value is either a number or a boolean. In the language above, values are both expressible and denotable. 

(define val? (or/c number? boolean?))
(define expressible? val?)
(define denotable? val?)

3.2. Environment

Environments are built by extending the empty environment. 

(define-datatype env env?
  [empty-env]
  [extended-env (syms (list-of symbol?))
                (vals (list-of val?))
                (outer-env env?)])

(define *empty-env* (empty-env))
(define *top-env* (extended-env '(x y z) '(2 4 6) *empty-env*))

4. Interpreter in Direct Style

4.1. Environment lookup

;;; lookup-env : [env? symbol?] -> denotable? + void?
(define (lookup-env e x)
  (cases env e
    [empty-env () (error 'lookup-env "~a not found~n" x)]
    [extended-env (syms vals outer-env)
                  (letrec ([loop (lambda (syms vals)
                                   (cond
                                     [(null? syms)
                                      (lookup-env outer-env x)]
                                     [(eq? (car syms) x)
                                      (car vals)]
                                     [else 
                                      (loop (cdr syms) (cdr vals))]))])
                    (loop syms vals))]))

;;; testing function
(define (lookup e x)
  (lookup-env e x))

4.2. Testing lookup

(check-equal? (lookup *top-env* 'x) 2)
(check-equal? (lookup *top-env* 'y) 4)
(check-exn exn:fail?
           (lambda () (lookup *top-env* 'w)))

4.3. Answer type

An answer is either an expressible value or void? which is the return type of error. 

(define answer? (or/c expressible? void?))

4.4. Evaluator

;;; eval-ast : [ast? env?]  -> answer?
(define eval-ast
  (lambda  (a e)
    (cases ast a
      [num (n) n]
      [bool (b) b]
      [id-ref (x) (lookup-env e x)]
      [ifte (test then else-ast)
            (let ([b (eval-ast test e)])
              (if (boolean? b)
                  (eval-ast (if b then else-ast) e)
                  (error 'eval-ast "~a not a boolean" b)))])))

(define (run exp)
  (eval-ast (parse exp) *top-env*))

4.5. Run tests

(check-equal? 5 (run '5))
(check-equal? 2 (run '(ifte #t x y)))
(check-equal? 2 (run 'x))
(check-equal? 4 (run '(ifte #f x (ifte #t y z))))

The interpreter also throws errors on some syntactically valid programs: 

(check-exn exn:fail? (lambda () (run 'w)))
(check-exn exn:fail? (lambda () (run '(if x y z))))

5. Interpreter in Continuation Passing Style

The interpreter in CPS now has continuation as part of its state.

5.1. Answer domain

An answer is now either an expressible value or error string. 

(define err-msg? string?)
(define answer? (or/c expressible? err-msg?))

5.2. Environment Lookup

The CPS version of lookup, called lookup/env, uses success and failure continuations. A failure continuation takes no arguments. A success continuation takes one argument. 

;;; succ? : val?  ->  answer?
;;; fail? : () -> answer?

;;; lookup-env/k : 
;;; [env? symbol? succ? fail?] -> answer?
(define (lookup-env/k e x succ fail)
  (cases env e
    [empty-env () (fail)]
    [extended-env (syms vals outer-env)
                  (letrec ([loop (lambda (syms vals)
                                   (cond
                                     [(null? syms)
                                      (lookup-env/k outer-env x succ fail)]
                                     [(eq? (car syms) x) 
                                      (succ (car vals))]
                                     [else 
                                      (loop (cdr syms) (cdr vals))]))])
                    (loop syms vals))]))

5.3. Testing lookup-env/k

;;; testing function
(define (lookup e x)
  (lookup-env/k e x (lambda  (v) v) (lambda  () (format "~a not found~n" x))))

(check-equal? (lookup *top-env* 'y) 4)
(check-equal? (lookup *top-env* 'w) 
              "w not found
")

5.4. Evaluator

The evaluator eval-ast/k (eval-ast with k) takes three arguments: an ast a, an environment e and a continuation k. *top-k* is the top level continuation. 

;;; k? : any/c? -> answer?
;;; eval-ast/k : ast? env? k? -> answer?
(define eval-ast/k
  (lambda  (a e k)
    (cases ast a
      [num (n) (k n)]
      [bool (b) (k b)]
      [id-ref (x) (lookup-env/k e x k
                                (lambda ()
                                  (*top-k*
                                   (format "~a not bound" x))))]
      [ifte (test then else-ast)
            (eval-ast/k test e
                        (lambda  (b)
                          (if (boolean? b)
                              (eval-ast/k (if b then else-ast) e k)
                              (*top-k*
                               (format "~a not a boolean" b)))))])))

;;; *top-k* : k?
(define *top-k*
  (lambda  (v)
    v))


;;; run: any/c -> (or/c answer? void?)
(define (run exp)
  (eval-ast/k (parse exp) *top-env* *top-k*))
